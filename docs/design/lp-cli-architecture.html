TL;DR + REST surface preview

Build a declarative, idiot-proof tool for managing operational state across all our apps (La Piazza, POS, Camper&Tour, ISOTTO, language tools, music player). Secrets rotation is the first concrete build. Architecture generalizes to AI model swaps, URL routing, feature flags, anything that changes at runtime.

Pattern: Resource + Provider + Test, driven by YAML, with kubectl-style context for ergonomics.

   YAML manifests  ──→  shared library (Resource + Provider + Test)
                          │
                          ├──→  CLI surface     (v1)  — Typer + Rich, sticky context
                          └──→  REST API surface (v2/v3 — paper-designed)

★ YAML-as-source pivot (unchanged from D2)

Pivot kept: YAML is the source of truth, the .env file is generated output. Single source, gitignorable, diff-able.
  • One YAML per app per env.
  • Lives off-repo at /opt/secrets/<app>/<env>.yaml, chmod 600 root.
  • <env>.previous.yaml kept for rollback.
  • Env codes locked: dev / stg / uat / prod.

Variable naming convention (future, after rebrand sweep): <APP>_<CATEGORY>_<KEY> → LAPIAZZA_DB_URL. v1 keeps existing prefixes.

★ kubectl-style Context Model (NEW in Draft 3)

With 3-5 apps × 4 envs = 12-20 contexts, you don't want --app lapiazza --env prod on every command. Set a sticky context, then run verbs against it.

How it works

# Set the current working context (sticky, written to ~/.lpr/current)
lpr ctx use lapiazza:prod

# All subsequent commands inherit it
lpr status
lpr list
lpr apply --secret db-password --dry-run

# See what you're on
lpr ctx current
  → lapiazza:prod  (set 2026-05-13 09:14, audit log: ~/.lpr/audit.log)

# List all known contexts (from ~/.lpr/config.toml)
lpr ctx list
  CONTEXT              HOST                  CONTAINER     LAST USED
  lapiazza:dev         localhost (docker)    borrowhood    2026-05-10
  lapiazza:stg         46.62.138.218         borrowhood    2026-05-12
* lapiazza:prod        46.62.138.218         borrowhood    2026-05-13
  pos:dev              localhost (docker)    pos-app       (never)
  camper:prod          46.62.138.218         camper-app    (never)

# Unset (forces explicit --ctx on next mutating command)
lpr ctx unset

# One-off override without changing sticky
lpr --ctx lapiazza:stg apply --dry-run

Why this matters

  • Ergonomic: match kubectl/helm mental model — every cloud-native operator already knows it.
  • Reduces typing: from lpr apply --app lapiazza --env prod --secret X to lpr apply --secret X.
  • State-aware: shell prompt can show current context (kube-ps1 equivalent).
  • Audit log gets richer: every entry records ctx, so log answers "what did I do on prod yesterday?" trivially.

Footgun risk + mitigations

Risk: sticky ctx is convenient but you can fire lpr apply thinking you're on staging when context says prod. Real damage potential.

Three mitigations, stackable (which subset → §17 Q-N1):

  • Banner mitigation: every command prints a 1-line banner at start with context, colored red if prod. Free.
  • Confirm mitigation: all mutating commands require --confirm (or LPR_PROD_OK=1 env) when ctx ends in :prod. (See §12.)
  • Shell-prompt integration: ship a lpr-ps1 snippet for .bashrc/.zshrc that shows ctx in the prompt. Optional but classy.

Default in Draft 3: banner ALWAYS + confirm on prod ALWAYS. Shell-prompt is opt-in.

3 Abstractions: Resource + Provider + Test (+ KC tie-in)

Resource

A thing managed declaratively. v1 implements only secret. Future: ai_model, route, feature_flag, quota, kc_realm, kc_client.

Provider

class SecretProvider:
    def current(name) -> str
    def apply(name, new_value)
    def test(name) -> bool
    def history(name) -> list

Test

Decoupled functions that verify state in the real world.

def test_asyncpg_connect_docker_network(env_file: Path) -> bool
def test_oidc_discovery_returns_200(realm_url: str) -> bool
def test_telegram_bot_getme(token: str) -> bool

Keycloak tie-in (from D2, unchanged)

  • KC client secrets (e.g., BH_OIDC_CLIENT_SECRET) — in v1 as regular secret resources. Rotation includes apply_extra: [kc_client_secret_rotate] recipe.
  • KC realm / user management — v2+ resource types (kc_realm, kc_user). Not v1.

KC realm naming (after rebrand sweep): <app>-<env> → lapiazza-prod, pos-prod, etc.

YAML Schema + timestamps (unchanged from D2)

Server-side, at /opt/secrets/<app>/<env>.yaml. Each secret carries:

  • last_rotated_at — ISO timestamp, written by tool on every successful apply.
  • rotation_interval_days — how often it should be rotated.
  • next_rotation_due — computed denormalization.
# /opt/secrets/lapiazza/prod.yaml
version: 1
app: lapiazza
env: prod
generated_env_file: /opt/helixnet/hetzner/lapiazza.env
app_container: borrowhood

secrets:
  - id: flask-secret-key
    env_var: BH_SECRET_KEY
    value: <actual value>
    last_rotated_at: 2026-02-15T10:30:00Z
    rotation_interval_days: 365
    next_rotation_due: 2027-02-15T10:30:00Z
    rotation:
      sources: [generate, paste, keepass]   # NEW in D3: keepass source
      generator: openssl_rand_base64_32
      keepass_entry: "LP-Production/BH_SECRET_KEY"
      test: app_healthz

  - id: db-password
    env_var: BH_DATABASE_URL
    composite:
      template: "postgresql+asyncpg://{user}:{password}@postgres:5432/{database}"
      parts: { user: lapiazza_app, password: <hex>, database: borrowhood }
    last_rotated_at: 2026-05-12T14:00:00Z
    rotation_interval_days: 365
    next_rotation_due: 2027-05-12T14:00:00Z
    rotation:
      sources: [generate, paste, keepass]
      generator: openssl_rand_hex_32
      apply_extra: [alter_user, kp_update]
      db_admin_role: helix_user
      keepass_entry: "LP-Production/BH_DATABASE_URL"
      test: asyncpg_connect_docker_network

NEW in Draft 3: each secret can carry keepass_entry path (consumed by pykeepass provider). If set, lpr apply can pull/push values from/to that KP entry.

Why TOML, not YAML, for the operator config: YAML is for declarative state (values), TOML is for configuration (settings). Standard Python ecosystem split. Pydantic loads both equally fine.

CLI Surface — sticky context (rewritten for D3)

Context commands (NEW)

  • lpr ctx use <app:env> — set sticky context; prompts master pw for KeePass
  • lpr ctx current — show what's set
  • lpr ctx list — table of all contexts from config.toml, * marks current
  • lpr ctx unset — clear sticky context

State queries (read-only — no --confirm needed)

  • lpr status — health, container state, overdue secrets, last apply
  • lpr list — secrets with last-rotated + next-due
  • lpr diff <other_env> — compare current ctx to another env of same app
  • lpr validate — schema check the manifest, no changes
  • lpr history <secret_id> — show rotation history from audit log

Mutating commands (require --confirm on prod)

  • lpr apply — apply manifest changes; interactive diff confirm
  • lpr apply --secret <id> — single-secret rotation
  • lpr apply --dry-run — show diff, no changes (no confirm needed)
  • lpr rollback — swap previous → current YAML, redeploy
  • lpr import --from /path/to/.env — bootstrap manifest from existing env file

Optional overrides (rare)

  • lpr --ctx <app:env> <cmd> — one-off override without changing sticky
  • lpr --no-banner — suppress context banner (for scripts)

Banner on every command

$ lpr status
[lapiazza:prod]  host=46.62.138.218  container=borrowhood
─────────────────────────────────────────────────────────
Manifest        OK
Container       Up 3 hours
Last apply      2026-05-12T14:00:00Z (db-password)
Overdue         0  |  Due in 30d: 0
─────────────────────────────────────────────────────────

Red color on the [lapiazza:prod] token when env is prod.

Audit Log + key fingerprints (NEW)

Every destructive op writes to ~/.lpr/audit.log with timestamp, context, action, key fingerprint (old → new).

~/.lpr/audit.log — one JSON per line, append-only.

{
  "ts": "2026-05-13T09:14:22Z",
  "ctx": "lapiazza:prod",
  "actor": "angel",
  "action": "rotate",
  "secret_id": "db-password",
  "source": "pykeepass:auto-generated",
  "fingerprint": {"old": "sha256:7a3f8b2c", "new": "sha256:e1d9a504"},
  "steps": ["edit_env", "pretest", "alter_user", "kp_update", "restart"],
  "test": "PASS",
  "outcome": "complete",
  "duration_s": 42
}

Why fingerprints, not values

  • Audit trail proves "the value changed" without ever logging the value itself.
  • If someone gets the log, they get nothing exploitable.
  • Fingerprint = first 8 chars of sha256(value). Collision probability negligible at this scale.

Query examples

# When did we last rotate db-password on prod?
$ lpr history db-password
2026-05-13T09:14:22Z  rotate  PASS  fingerprint 7a3f...→e1d9...
2026-05-12T14:00:00Z  rotate  FAIL  (rolled back, env mismatch)
2026-02-15T10:30:00Z  rotate  PASS  fingerprint 9b2c...→7a3f...

# Anything failed across all contexts in the last week?
$ lpr history --failures --since 7d

Safety mechanisms + prod guardrails (expanded)

Core safety (kept from D2)

  1. Backup before any state-changing step (timestamped previous.yaml)
  2. Pre-test BEFORE state change (idempotent if already correct)
  3. Re-test AFTER state change (rotation only "complete" if test passes)
  4. Auto-rollback on failure (restore backup, restart with old creds)
  5. Hidden input for any manual paste
  6. Compare-paste enforcement (when source=paste)
  7. Append-only audit log, including failures

NEW in Draft 3 — prod-context guardrails

lpr rotate and lpr deploy require --confirm when context is prod, OR an env var LPR_PROD_OK=1 set for the session.
  1. --confirm required on prod context. Every mutating command (apply, rollback, import) refuses without it when ctx ends in :prod.
  2. LPR_PROD_OK=1 env escape. Set in shell once → suppresses --confirm requirement for that session. Used for CI / scripted ops only. Banner stays red.
  3. Red banner ALWAYS on prod context, regardless of confirm state.
  4. Context-mismatch abort: if lpr apply --secret X is run, and the manifest says ctx app:prod but the sticky ctx says app:stg, abort with clear error.

Example flows

# Day-to-day on dev (no friction)
$ lpr ctx use lapiazza:dev
$ lpr apply --secret flask-secret-key
[lapiazza:dev]  ← green banner
Diff:
  flask-secret-key:  abc... → xyz...
Apply? [y/N]: y
✓ rotated, test passed, audit logged.

# Prod (friction-on)
$ lpr ctx use lapiazza:prod
$ lpr apply --secret db-password
[lapiazza:prod]  ← red banner
✗ ABORT: prod context requires --confirm or LPR_PROD_OK=1

$ lpr apply --secret db-password --confirm
[lapiazza:prod]  ← red banner
Diff: ...
Apply to PROD? type 'YES' to confirm: YES
✓ rotated.

KeePass — pykeepass in v1 (rewritten)

pykeepass, not shelling out. Lets `lpr rotate jwt` generate a new key, write it to the right entry in the .kdbx, push it to the env on Hetzner, restart the container, and roll back if health check fails — all in one atomic-ish flow.

Draft 2 had a v1 → v4 ladder (manual paste → keepassxc-cli → write back → hands-free). Draft 3 jumps the ladder: v1 = pykeepass with read+write from day one.

Library: pykeepass

  • Pure Python, no subprocess, no system dependency
  • KDBX 3.x and 4.x compatible
  • Open with master password (prompted by Typer)
  • Read attribute via path: kp.find_entries(path=['LP-Production', 'BH_SECRET_KEY'])
  • Set password, save back
  • Attach files (for periodic YAML backup)

v1 flow

$ lpr ctx use lapiazza:prod
Master password for ~/Documents/keepass/main.kdbx: ********
✓ KeePass DB unlocked (cached for this session).

$ lpr apply --secret db-password --confirm
[lapiazza:prod]
Source: generate (will write new value to KP + server)
Generating: openssl_rand_hex_32
Diff:
  db-password: sha256:7a3f... → sha256:e1d9...
Steps that will run:
  1. Write new value to KeePass entry "LP-Production/BH_DATABASE_URL"
  2. ALTER USER lapiazza_app on Postgres
  3. Update /opt/secrets/lapiazza/prod.yaml (server)
  4. Regenerate /opt/helixnet/hetzner/lapiazza.env
  5. Restart borrowhood container
  6. Health check
  (If anything fails: roll back env + Postgres password + KP entry)

Apply to PROD? type 'YES' to confirm: YES
✓ All steps green. Audit logged.

Master password caching

Within one shell session: the unlocked KeePass DB is held in memory until lpr ctx unset or shell exits. No re-prompting between commands.

Across sessions: always re-prompts. Master pw never written to disk.

When prompted: open question §17 Q-N3.

TL;DR

Build a declarative, idiot-proof CLI for managing La Piazza's operational state. The secrets rotation tool is the first concrete build — but the architecture must generalize so the same pattern eventually handles AI model swaps, payment provider switches, URL routing changes, feature flags, anything the platform exposes that needs to change at runtime.

Pattern: Resource + Provider + Test, driven by YAML.

   declarative YAML ──→ CLI (interactive, paranoid) ──→ apply changes ──→ verify
                                  │
                                  └──→ eventually wrapped as REST API
                                       (same logic, different surface)
2

★ Design Pivot — YAML is source, env is generated output

Original design: the tool edits borrowhood.env in place. Env file is BOTH source of truth and runtime config.
Pivoted design: the env file is a generated output, not a source of truth. The source is a YAML file (per environment) that lives OFF the git repo. The tool reads YAML → generates env file → deploys.

Why the pivot is better

  • Single source of truth (the YAML). No drift between human-edited file and runtime config.
  • Architecturally impossible to commit secrets to git (env files are generated, gitignored; YAML lives outside repo).
  • Per-environment files (prod.yaml, staging.yaml, future preprod.yaml, uat.yaml) are the same shape.
  • Rollback is symmetric: prod.previous.yaml always exists as the last working version.
  • Diff-able and dry-runnable.

Why this isn't overkill

Industry has bigger off-the-shelf options. Honest map:

ToolWhat it doesRight for us?
HashiCorp VaultCentralized secret store, audit, rotationOverkill — needs ops effort to run/secure
Mozilla sopsEncrypts YAML at rest with age/PGP keysEasy v2 upgrade for us
Doppler / InfisicalSaaS secret manager$10-30/mo per dev, external dependency
AWS Secrets ManagerCloud-nativeWe're on Hetzner
k8s Secrets + ESODeclarative k8s patternWe use docker-compose
Manual nano editsWhere solo-dev mistakes liveWhat we did until today — broken

Verdict: not reinventing the wheel — building the right size wheel for our scale. v2 with sops/age gets us to "production-grade for small team" without throwing away anything.

The 3 Abstractions: Resource + Provider + Test

Resource

A thing managed declaratively. v1 implements only secret. Future: ai_model, route, feature_flag, quota.

Provider

Code that knows how to read, write, test, and rotate a specific resource type. One provider per resource type.

class SecretProvider:
    def current(name) -> str         # read current value
    def apply(name, new_value)       # set new value
    def test(name) -> bool           # verify it works
    def history(name) -> list        # rotation log

Test

Small functions that verify state. Decoupled from Providers so they can be reused.

def test_asyncpg_connect_docker_network(env_file: Path) -> bool
def test_oidc_discovery_returns_200(realm_url: str) -> bool
def test_telegram_bot_getme(token: str) -> bool

The YAML Schema (post-pivot)

Location: NOT in the git repo. Lives at /opt/helixnet/secrets/<env>.yaml on Hetzner, chmod 700 on directory, 600 on each file, owned by root.

One YAML per environment. Rollback pair: before any apply, current snapshots to <env>.previous.yaml.

# /opt/helixnet/secrets/prod.yaml
version: 1
env_name: prod
generated_env_file: /opt/helixnet/hetzner/lapiazza.env
app_container: borrowhood
hetzner_host: 46.62.138.218

secrets:
  - id: flask-secret-key
    env_var: BH_SECRET_KEY
    value: <actual value>
    rotation:
      sources: [generate, paste]
      generator: openssl_rand_base64_32
      test: app_healthz

  - id: db-password
    env_var: BH_DATABASE_URL
    composite:
      template: "postgresql+asyncpg://{user}:{password}@postgres:5432/{database}"
      parts:
        user: lapiazza_app
        password: <hex value>
        database: borrowhood
    rotation:
      sources: [generate, paste]
      generator: openssl_rand_hex_32
      apply_extra: [alter_user]
      db_admin_role: helix_user
      test: asyncpg_connect_docker_network

  # ... 7 more secret entries

CLI Surface

Modes

  • lp-rotate — fully interactive (menu walk)
  • lp-rotate apply --env prod --secret db-password — single-secret targeted
  • lp-rotate apply --env prod --dry-run — show diff, no changes
  • lp-rotate list --env prod — show all secrets with last-rotated dates
  • lp-rotate status — overdue/incomplete summary
  • lp-rotate rollback --env prod — swap previous → current, redeploy
  • lp-rotate validate --env prod — schema check, no changes

Directory Structure

# OFF-REPO (server only, chmod 700, root):
/opt/helixnet/secrets/
├── prod.yaml                    ← source of truth
├── prod.previous.yaml           ← rollback target
├── staging.yaml
└── staging.previous.yaml

# IN-REPO (the tool):
helixnet/scripts/
├── lp_rotate.py                 ← Typer CLI entry
└── lp/
    ├── cli.py
    ├── manifest.py
    ├── providers/secret_provider.py
    ├── recipes/{edit_env,alter_user,restart_container}.py
    ├── tests/{asyncpg,oidc,telegram,paypal,resend,ollama,healthz}.py
    ├── generators.py
    ├── audit.py
    └── ssh.py

# GENERATED (server, gitignored):
/opt/helixnet/hetzner/
└── lapiazza.env                 ← from prod.yaml
Rule: nothing in /opt/helixnet/secrets/ ever touches git. Nothing in git ever contains real secret values.

Audit Log Format

~/.lp-rotation-log.jsonl — one JSON per line, append-only:

{"ts":"2026-05-13T08:14:22Z","env":"prod","secret_id":"db-password","actor":"angel","source":"paste","steps":["edit_env","pretest","alter_user","restart"],"test":"PASS","outcome":"complete","duration_s":42}

Query examples: "when did we last rotate X?" → grep + tail. "Anything failed lately?" → grep failed. Five lines of code, immense future value.

10

Safety Mechanisms (Paranoia in Code, Not Discipline)

  1. Backup before any state-changing step (timestamped)
  2. Pre-test BEFORE state change (idempotent if already correct)
  3. Re-test AFTER state change (rotation only "complete" if test passes)
  4. Auto-rollback on failure (restore backup, restart with old creds)
  5. Hidden input (read -s style) for password paste
  6. Compare-paste enforcement: prompt twice, strcmp, mismatch = abort
  7. KeePass as single source of truth (tool never tries to store)
  8. Append-only audit log of every action including failures

Extra questions (last-pass before code)

  1. YAML seed/import tool — first run needs to bootstrap prod.yaml from existing env. Add lp-rotate import --env prod --from /opt/helixnet/hetzner/borrowhood.env? My lean: YES, ~30 LOC.

  4. Validate-only mode — lp-rotate validate --env prod checks YAML schema without applying. Yes, ~10 LOC.
  5. Staging-before-prod soft check — warn (not block) if no recent staging apply. Yes, ~20 LOC.

  • lp-rotate list shows all secrets
  • --dry-run shows diff without changes
  • apply works end-to-end with auto-rollback on test failure

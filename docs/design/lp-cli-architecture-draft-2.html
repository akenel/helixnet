TL;DR + REST surface preview

Build a declarative, idiot-proof tool for managing operational state across all our apps (La Piazza, POS, Camper&Tour, ISOTTO, language tools, music player). Secrets rotation is the first concrete build. Architecture generalizes to AI model swaps, URL routing, feature flags, anything that changes at runtime.

Pattern: Resource + Provider + Test, driven by YAML.

   YAML  ──→  shared library (Resource + Provider + Test)
              │
              ├──→  CLI surface     (v1)     # lpr apply --app lapiazza --env prod
              └──→  REST API surface (v2/v3) # POST /apps/lapiazza/envs/prod/apply
same logic keep it simple to get the idea of routes URLs for each CLI command maybe think REST API first but skip CLI .. yagni

How I read it: design the REST surface on paper alongside each CLI command (1:1 mapping), so v2 wrap is mechanical. But build the CLI first because that's what one wolf at a kitchen table can use today. The shared library is the actual thing — CLI & REST are skins.

CLI command (v1)REST equivalent (v2/v3 paper-only for now)
lpr list --app lapiazza --env prodGET /apps/lapiazza/envs/prod/secrets
lpr apply --app lapiazza --env prodPOST /apps/lapiazza/envs/prod/apply
lpr apply --dry-runPOST /apps/lapiazza/envs/prod/apply?dry_run=true
lpr rollback --env prodPOST /apps/lapiazza/envs/prod/rollback
lpr diff prod stagingGET /apps/lapiazza/diff?a=prod&b=staging
lpr validate --env prodPOST /apps/lapiazza/envs/prod/validate
lpr import --env prod --from .envPOST /apps/lapiazza/envs/prod/import
lpr statusGET /status

v1 commitment: CLI only. Paper commitment: every command maps cleanly to one REST route. So v2 wrap = wiring + auth, not redesign.

Why the YAML pivot is better (unchanged from Draft 1)

  • Single source of truth (the YAML).
  • Architecturally impossible to commit secrets to git.
  • Per-environment, per-app files all share the same shape.
  • Rollback is symmetric.
  • Diff-able and dry-runnable.

The 3 Abstractions: Resource + Provider + Test (+ KC realm note)

Resource

A thing managed declaratively. v1 implements only secret. Future: ai_model, route, feature_flag, quota, kc_realm, kc_client.

Provider

class SecretProvider:
    def current(name) -> str         # read current value
    def apply(name, new_value)       # set new value
    def test(name) -> bool           # verify it works
    def history(name) -> list        # rotation log

Test

def test_asyncpg_connect_docker_network(env_file: Path) -> bool
def test_oidc_discovery_returns_200(realm_url: str) -> bool
def test_telegram_bot_getme(token: str) -> bool

YAML Schema + timestamps (NEW)

Location: NOT in the git repo. Lives at /opt/secrets/<app>/<env>.yaml on Hetzner, chmod 700 dir, 600 each file, owned by root.

One YAML per app per environment.

NEW in Draft 2 — Rotation timestamps

we need the next rotation dates or some date and time info — like last changed or something

Three new fields per secret, all optional (tool sets last_rotated_at automatically on apply):

  • last_rotated_at — ISO timestamp, written by tool on every successful apply
  • rotation_interval_days — how often the secret should be rotated (e.g., 365 for annual, 90 for quarterly)
  • next_rotation_due — computed by tool = last_rotated_at + rotation_interval_days (denormalized for easy human read)

lpr status queries these and shows red rows for overdue, yellow for due-soon.

Full schema example

# /opt/secrets/lapiazza/prod.yaml
version: 1
app: lapiazza
env: prod
generated_env_file: /opt/helixnet/hetzner/lapiazza.env
app_container: borrowhood          # docker compose service name
hetzner_host: 46.62.138.218

secrets:

    env_var: BH_SECRET_KEY
    value: <actual value>
    last_rotated_at: 2026-02-15T10:30:00Z
    rotation_interval_days: 365
    next_rotation_due: 2027-02-15T10:30:00Z

      sources: [generate, paste]
      generator: openssl_rand_base64_32
      test: app_healthz

  - id: db-password
    env_var: BH_DATABASE_URL

      template: "postgresql+asyncpg://{user}:{password}@postgres:5432/{database}"

        user: lapiazza_app
        password: <hex value>
        database: borrowhood
    last_rotated_at: 2026-05-12T14:00:00Z
    rotation_interval_days: 365
    next_rotation_due: 2027-05-12T14:00:00Z

      sources: [generate, paste]
      generator: openssl_rand_hex_32
      apply_extra: [alter_user]

CLI binary renamed: lp-rotate → lpr everywhere. Python file: scripts/lpr.py.

Modes (Draft 2)

  • lpr — fully interactive (menu walk; asks for app, env, action)
  • lpr apply --app lapiazza --env prod --secret db-password — single-secret targeted
  • lpr apply --app lapiazza --env prod --dry-run — show diff, no changes
  • lpr list --app lapiazza --env prod — show all secrets with last-rotated + next-due dates
  • lpr status — overdue/incomplete summary across ALL apps + envs
  • lpr rollback --app lapiazza --env prod — swap previous → current, redeploy
  • lpr validate --app lapiazza --env prod — schema check, no changes
  • lpr diff --app lapiazza prod staging — multi-env diff (was v2, now v1)
  • lpr import --app lapiazza --env prod --from /opt/helixnet/hetzner/borrowhood.env — bootstrap YAML from existing env

Default app: if --app omitted, prompts interactively (or reads ~/.lpr.toml for a default).

Audit Log Format

~/.lpr-log.jsonl — one JSON per line, append-only:

{"ts":"2026-05-13T08:14:22Z","app":"lapiazza","env":"prod","secret_id":"db-password","actor":"angel","source":"paste","steps":["edit_env","pretest","alter_user","restart"],"test":"PASS","outcome":"complete","duration_s":42}
does this null out my concerns that we have not dates in the yamls to identify edit times and logging details of changes and diffs etc

Short answer: NO, they complement each other.

  • YAML timestamps = current state snapshot ("when was X last rotated, when is next due"). Easy human read.
  • Audit log = full history with diffs, who/what/when, including failures. The forensic trail.

YAML answers "what's the state now?" — log answers "what happened over time?". Both are cheap to have.

Safety Mechanisms + KeePass roadmap

  1. Backup before any state-changing step (timestamped)
  2. Pre-test BEFORE state change (idempotent if already correct)
  3. Re-test AFTER state change (rotation only "complete" if test passes)
  4. Auto-rollback on failure (restore backup, restart with old creds)
  5. Hidden input (read -s style) for password paste
  6. Compare-paste enforcement: prompt twice, strcmp, mismatch = abort
  7. KeePass as single source of truth (tool never tries to store)
  8. Append-only audit log of every action including failures

eventually have the v2 keepass api working to assist and make it hands free and just done or error etc occurs maybe v2.4 or v3 or v4 unless its super easy would and should be the end goal
VersionKeePass integration level
v1Manual paste from KeePass. Tool prompts, you copy from KP, paste hidden.
v1.5Manual YAML backup attached to KeePass entry as periodic ritual (your sec-D ask).
v2Read from KeePass via keepassxc-cli wrapper — semi-hands-free (still need master password unlock).

v4Full hands-free: scheduled rotation, KeePass sync, alerts on overdue.

End goal: zero manual paste. But every version stays usable on its own.

NEW in Draft 2 — Your scrambling question

since we are asking for a pw should we not scramble it so what you see in the yaml is not the actual pw but a derived value of kind of a algorithm is applied and the actual hex value [is recovered at runtime]

This is the right instinct. You're describing encryption at rest. Two paths:

ApproachHowTrade-off
v1 — chmod + offlineYAML lives off-repo, chmod 600 root, never leaves HetznerSimple. Plain text if you SSH in as root. Adequate for solo + 1 server.
v1.5 — sops + ageEncrypt YAML at rest with age key. Tool decrypts in-memory only.~50 LOC addition. Industry standard. You unlock once per session with age private key.
"Custom scramble"Hand-rolled obfuscation algorithmDon't do this. Either it's a real cipher (use sops) or it's security theater. No middle ground.

My lean: v1 chmod 600 root only, v1.5 sops/age before we have 5+ apps × 4 envs = ~30+ files of secrets sitting in plain text. That's the "real prod" gate.

Open question in section 16 for your call: should v1 ship with sops/age built-in, or strictly chmod 600?

What you're doing: the database backups already run on Hetzner at 3am, every night, with zero dependency on this laptop — dump → encrypt → self-verify restore → 30-day keep → immutable copy to Backblaze B2. This runbook bolts on the last two things that make it bullet-proof:

🔔 An alarm (healthchecks.io "dead-man's-switch") that screams to your email/Telegram the moment a backup silently fails or the box goes dark.   🔑 A restore key stored in KeePass so you can actually pull a backup out of B2 in a disaster (the box's key is write-only — it can save but not restore).

🧍 YOU do the clicks 🐯 TIGS wires the box + fires the test SETUP — do it once
🎯 The one handoff: at the end of Part 1 you copy a Ping URL and give it to Tigs. That's the only thing Tigs needs — everything else you keep (the read-key goes straight into KeePass, never into chat).

1 · The alarm — healthchecks.io you

A "dead-man's-switch": the box pings this URL every night when the backup succeeds. If a ping ever doesn't arrive (backup failed, box down, cron gone) — it alerts you. That's the only thing that catches a silent failure.

Open healthchecks.io ↗ healthchecks.io — free, 20 checks
  1. Sign up at healthchecks.io (email, Google, or GitHub — free). you land on the My Checks dashboard
  2. Click Add Check. Rename it (pencil icon) to banco-nightly-backup. a check card appears with a unique Ping URL like https://hc-ping.com/xxxxxxxx
  3. Open the check → Schedule → set Period = 1 day, Grace = 2 hours. Save. reads "expected every 1 day, grace 2 h" — backup runs 3am, so if no ping by ~5am it alarms
  4. Go to Integrations → add Email (and/or Telegram: click Telegram → open the bot link → /start → connect your chat). Hit Send Test Notification. a test alert lands in your inbox / Telegram — confirms the alarm can reach you
  5. Copy the check's Ping URL (the https://hc-ping.com/… one). it's on your clipboard
  6. Test it yourself: in a terminal run curl «your ping URL». the check flips to green / "up" on the dashboard within seconds — you just proved the ping works
  7. 🎯 Hand the Ping URL to Tigs (paste it in chat — a ping URL is low-risk; worst case someone could only suppress an alarm, never read data). Tigs wires the box's 3am job to ping it on success and /fail on failure

2 · The restore key — B2 read-only you

The box uploads with a write-only key (can save, can't read — good). But to restore in a disaster you need a read key, and it belongs in KeePass, not on any server.

Open B2 → Application Keys ↗ secure.backblaze.com → Application Keys
  1. Add a New Application Key. the key-creation form opens
  2. Name banco-restore-readonly; Allow access to Bucket(s) = ecolution-banco-backups (scope it); Type of Access = Read Only. Create. it shows a keyID and an applicationKey — the applicationKey appears only once
  3. Save both into KeePass as a new entry: B2 banco restore (read-only) — keyID in the username field, applicationKey in the password field. saved in your kdbx (which syncs to Drive) — the restore key now lives where DR expects it
  4. (Optional, with Tigs) prove it reads: use the read-key to download the newest B2 blob. it downloads + decrypts — the offsite copy is provably recoverable by you

3 · Your KeePass recovery key-ring is complete you — 30-sec check

The "can I actually recover?" checklist. These four together = a full restore, even if this laptop is gone.

  1. KeePass master password — in your head (unlocks the whole kdbx). you know it — this is the one thing that isn't written down anywhere
  2. GPG backup passphrase — decrypts the .gpg blobs (the box's /root/.banco-backup-key). already an entry in KeePass (you decrypted with it on 2026-07-01)
  3. B2 restore read-key — just added in Part 2. present in KeePass
  4. B2 master key — your admin record. saved in KeePass (or you can regenerate it in the B2 console anytime)
🧭 The whole recovery, in one line: unlock KeePass → grab the read-key → download the newest blob from B2 → decrypt with the GPG passphrase → restore into Postgres. Nothing on this laptop is required.

4 · Tigs wires it + we fire a LIVE test tigs, after your handoff

The proof. A backup alarm you haven't watched fire is just a hope.

  1. You gave Tigs the Ping URL → Tigs adds it to the box's banco_backup.sh: ping on success, /fail on any error. Tigs confirms the hook is in + the nightly job will ping from tonight
  2. Live success test: Tigs runs the backup once by hand. the healthchecks dashboard shows a fresh ping + the check is green "up"
  3. Live FAILURE test: Tigs simulates a failed backup (e.g. bad key path). within the grace window you get a real alert (email/Telegram) — this is the whole point, watch it fire
  4. Restore proof: Tigs (or you) uses the read-key to pull today's blob from B2 and decrypt it. it restores — end-to-end, offsite copy is genuinely recoverable

★ Done when

  1. Alarm is live and you watched a test alert arrive — a silent failure can never happen again.
  2. Restore read-key is in KeePass + the key-ring is complete — you can recover from B2 with nothing but your KeePass password.

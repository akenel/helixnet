The scenario: someone deletes two important records. We look for them — gone. Panic. "We have a big problem." So we restore the database from a backup taken an hour ago and get them back. That's the whole point of every backup we built — this proves it actually works when it counts.

The two victims (546 products in sandbox; delete these → 544 → restore → 546):
Victim 1 · 4-part Grinder 50mm
SKU ACC-GRINDER · CHF 15.00
Victim 2 · Grinder Kräutermühle Tabakmühle 2-teilig
SKU SEED-B0BCB0B273 · CHF 23.95
SANDBOX only — prod is never touched 🐯 Tigs runs the DB ops 🧍 You eyeball the till + call it
🛟 Safety net: this is sandbox — if anything goes sideways, make sandbox-reset rebuilds it from seed. There is no way to hurt real data here. That's why we rehearse disasters in sandbox, never prod.

0 · Setup — sandbox can back up to B2 tigs, once

Same machinery as prod, pointed at banco_sandbox → encrypted → the B2 vault (sandbox/ prefix). This is your practice run of the whole backup rig.

  1. Tigs installs a sandbox backup (banco_sandbox_backup.sh) + does a dry run: back up → restore into a throwaway DB → row counts match. the sandbox backup rig round-trips clean BEFORE we touch a single real step — facts straight

1 · Confirm the two records EXIST you

Before the disaster, prove they're there — so "gone" later means something.

Open the sandbox till ↗
  1. On the sandbox till, search grinder. Find both victims: 4-part Grinder 50mm (CHF 15) and Grinder Kräutermühle 2-teilig (CHF 23.95). both products show up, sellable
  2. Tigs confirms the count at the DB level. 546 products total; both victim SKUs present

2 · Take the "good" backup tigs

This is the snapshot we'll fall back to — "the database as it was an hour ago." It has all 546, including both victims.

  1. Tigs runs the sandbox backup NOW → encrypts → ships to the B2 vault (immutable). a sandbox backup file is made (546 products) + lands in B2 with an object-lock date

3 · 💥 THE DISASTER — delete the two records destructive · sandbox

The "oops." Two records get deleted (as if a bad script, a fat finger, or a bad actor did it).

  1. Tigs deletes ACC-GRINDER and SEED-B0BCB0B273 from banco_sandbox. 2 rows deleted → 544 products remain

4 · 😱 Confirm they're GONE you — the panic moment

This is what a real incident feels like: the record isn't there and everyone's asking where it went.

  1. On the sandbox till, search grinder again. the two victims are GONE — the 4-part Grinder 50mm and the Kräutermühle 2-teilig are missing from the results
  2. The gap is measurable: the backup has 546, the live DB has 544. the difference (2) is exactly our lost records — proof the backup holds what the live DB lost

5 · 🔧 RESTORE from the backup tigs

The rescue: pull the good backup out of the B2 vault, decrypt it, and reload banco_sandbox — exactly the DR-brief procedure.

  1. Tigs downloads the good backup from B2 → decrypts (GPG passphrase) → reloads it into banco_sandbox (drop → recreate → load). the restore runs clean; sandbox is now the "before-disaster" state

6 · 🎉 Confirm they're BACK you — the relief

The payoff: the "lost" data is recovered, and you can see it with your own eyes.

  1. On the sandbox till, search grinder one more time. both victims are BACK — 4-part Grinder 50mm (CHF 15) and Kräutermühle 2-teilig (CHF 23.95), sellable again
  2. Count check. 546 products again — the exact number from before the disaster

7 · Sanity — nothing real was ever at risk no regression

  1. Prod was never touched — the whole drill was banco_sandbox. prod product count unchanged (5,200); prod never connected to
  2. The safety net was always there. make sandbox-reset would have rebuilt sandbox too — belt and suspenders

What this tests: the seal sweep (BL-100 #4) — a read-only, one-click integrity check inside the Hypercare Cockpit that hunts barcode cross-wires so we find a bad match here instead of at the till (the most expensive place to find it). The lesson: "if one seal fails, check all the seals." It runs two checks that map to the two real till incidents:

🔴 Collision — one barcode wired to more than one product (the GIZEH 42470335 that was on both Pink Rolls and Black Rolls).   🟠 Stranded — a discontinued row still holds a scan-live code while a live same-name item still sells (the Tycoon Gas "discontinued" dead-end).

It never changes data — it only SURFACES findings for you to confirm physically, then fix through the gated path.

Login: felix (admin) or any manager — the cockpit is manager/admin only. SANDBOX only — no prod machine-green ≠ human-green — your sign-off
🌱 A demo collision is seeded on sandbox so a finding card actually renders: barcode 42238072 is wired to GIZEH Black Rolls Slim (primary) + GIZEH Active Filter 6mm (alias). Clear it whenever you're done: DELETE FROM product_barcodes WHERE barcode='42238072' AND product_id='5ef90b70-80d6-40e5-b4d5-0e1932099ce2'; (banco_sandbox only).

0 · Open the cockpit manager/admin

The sweep lives as a panel at the top of the Hypercare Cockpit, above the feedback queue.

Open the Cockpit (sandbox) ↗ /pos/hypercare
  1. Log in as felix (or a manager) and open /pos/hypercare. the cockpit loads; the scorecard + feedback queue are there as before
  2. Look at the top, above the ticket filters, for the "🔒 Product-matching integrity" card. a card with a subtitle "The seal sweep — barcode cross-wires caught here instead of at the till" and a 🔍 Run seal sweep button on the right

1 · Run the sweep the one click

One button. It queries every barcode (primary + aliases) and reports the cross-wires — read-only, nothing changes.

  1. Click 🔍 Run seal sweep. the button shows "Sweeping…" briefly, then results appear below it; a toast reports the count (e.g. "1 cross-wire to review")
  2. Nothing about the catalog changed — this is a read-only report. no product was edited; you can run it again and again safely

2 · Read the collision 🔴 the real catch

The high-severity check: one barcode ringing up different items. The seeded demo mirrors the real GIZEH case.

  1. Find the red "Barcode on more than one product" block with a count. shows · 1 (the seeded demo) with a red 🔴 marker and a one-line hint about confirming which product owns the code
  2. Read the finding card for barcode 42238072. "📟 42238072 — wired to 2 products" then two rows
  3. Check the two rows name the products + their role. GIZEH Black Rolls Slim tagged primary and GIZEH Active Filter 6mm 34pcs tagged alias (SKUs + price shown too)
  4. Sanity: this is exactly what you'd need to fix it — you can see which code is double-booked and on which two items. enough to walk to the shelf, scan the real box, and know which row keeps the code

3 · Stranded check stays quiet no false alarms

Sandbox has 100+ discontinued items that still carry a barcode — but that's NORMAL, not a trap. The check only fires when a live same-name item exists (the Tycoon shape). So it should read 0 here — no wall of noise.

  1. Look for the amber "Discontinued row holds a code, but a live item has the same name" block. it is absent / 0 on sandbox — you do NOT see a huge list of every discontinued item
  2. This is deliberate — a retired item scanning as "discontinued" is correct, not a trap. the tool stays quiet unless there's a genuine live-twin conflict; signal, not noise
